## 1. Phase 3 mission

**Telegram MVP, end-to-end, production-shaped**: ship the smallest
slice that lets a real user issue a prompt over Telegram, watch
streaming output, click an approval button, and have the daemon stay
up across codex restarts and Mac mini reboots — without any test rig
in the call path.

Concretely, that bundles five interdependent slivers (deliberately
narrow scopes; nothing in this list pulls in CU, Lark, DingTalk, or
Web Console):

1. **Production daemon wire-up** (`@codex-im/daemon`) — replace the
   test-only `phase2-e2e-rig.ts` daemon-wireup function with a
   production module. The v2 subscription order is the **two-phase
   callback token flow** (D33), NOT the v1 `sendCard → bindActorPolicy`
   ordering (which the v1 review rejected as racy):
   `broker.onPendingCreated → SecurityPolicy.checkApprovalDestination
    → daemon issues opaque callback token + bindActorPolicy
    → projectAsRichBlock + render with token in callback_data
    → adapter.sendCard → daemon attaches MessageRef to token record`,
   and inbound: `adapter.onAction → token-record lookup +
   messageRef validation → SecurityPolicy.checkUserAndChat
   → broker.resolve → adapter.answerAction → adapter.updateCard`.
   Plus a CommandRouter and a SessionRouter (chat ⇄ project ⇄ thread).
2. **SecurityPolicy ACL** (`@codex-im/core`) — Phase 2 left
   `SecurityPolicy` as a `phase1-noop` mention in `types.ts`. Phase 3
   ships a real `SecurityPolicy` that runs BEFORE `bindActorPolicy`
   (so unauthorized chats never see the buttons) AND runs again
   inside the broker → resolve path (defense in depth). Includes
   user/chat/project allowlists, command deny patterns, and the
   "unknown method ⇒ critical, decline-only" defensive renderer
   path that's already partially implemented.
3. **`@codex-im/im-telegram`** (new package) — real Telegram
   ChannelAdapter built on `grammY`, conforming to the closed
   `ChannelAdapter` interface verbatim. Mirrors the constraints
   `TelegramShapeFakeChannelAdapter` already enforces (callback_data
   ≤62B, 60s answer-callback-query deadline, no parse_mode reliance).
4. **Persistent SessionRouter** (`@codex-im/core` + new
   `@codex-im/storage-sqlite`) — chat→project→thread bindings stored
   in SQLite so the daemon can survive a restart with bindings intact
   (a P0 acceptance criterion in 09-ROADMAP §Phase 2).
5. **launchd integration** (operational, in `packages/daemon` or a
   dedicated `packages/launchd`) — installable plist + log rotation
   wiring so the Mac mini "always-on daemon" promise is actually
   keepable.

**Why bundle them.** Each one in isolation is a half-feature: a
daemon wire-up with no Telegram has nothing to wire up; Telegram
without SecurityPolicy is a broken-by-design open relay; Telegram
with no SQLite means every codex restart loses bindings; without
launchd none of it survives a reboot. The bundle is the smallest
slice that lets Phase 4 (Lark) start cleanly.

**What is explicitly NOT in this mission**: see §3 Non-goals.

## 3. Goals and Non-goals

### 3.1 Goals (P0 — must ship)

- G1 **End-to-end Telegram-driven turn**: user sends text in a Telegram
  chat the daemon is configured to listen on; daemon starts a Codex
  thread (or reuses the bound thread); streams agent output back; if
  the model requests command execution / file change / permissions,
  daemon renders the approval card with inline keyboard; user clicks
  allow/decline; broker resolves; codex sees the decision; turn
  completes; daemon edits the streaming message with the final
  outcome.
- G2 **SecurityPolicy fail-closed gating**: an unauthorized
  user/chat/project combination cannot trigger a turn (inbound message
  dropped) AND cannot resolve an approval (broker's `bindActorPolicy`
  is never called for unauthorized targets, so any inbound action
  hits `binding_required`).
- G3 **Daemon restart preserves bindings**: chat→project→thread
  bindings live in SQLite (WAL); after `pkill -SIGTERM daemon` the
  next start picks up where it left off.
- G4 **launchd-installable**: a `bin/` script (or `pnpm run
  launchd:install`) installs a plist; `pnpm run launchd:uninstall`
  removes it; logs rotate; restart-on-crash works.
- G5 **No regression on Phase 2 internal gates**:
  typecheck/test/lint/method-literal-boundary/F13 channel-core
  boundary all stay green.
- G6 **Telegram callback_data within real platform constraints**:
  encoding via D30 (v2) opaque token (`"v1:" + 16-char base32`,
  ≤19 bytes total — well under Telegram's 64B limit). `${approvalId}|
  ${kind}|${nonce}` (v1) is rejected; see D30 / D34.
- G7 **Token redaction**: bot token never appears in any audit event
  payload, log line, thrown error message, plist file, SQLite column,
  or fixture.
- G8 **Synthesized `turn_failed` events on transport-loss** (Phase 1
  supervisor TODO; promoted P1→P0 per v2 Lake review). Without this,
  the IM layer sees no terminal event when codex restarts mid-turn —
  user UX silently stalls.
- G9 **Lazy-prune sweep for terminal `#pendingById` records** (P1→P0
  per v2 Lake review). Without this, long-running daemons leak memory
  proportional to total approvals served.

### 3.2 Goals (P1 — ship if cheap, defer otherwise)

- G10 `/status`, `/projects`, `/use`, `/new`, `/stop` slash commands
  for the Telegram chat (the minimum command set listed in 01-PRD §6).
  G1 only requires inbound prompt → outbound reply; the slash commands
  are operationally useful but the e2e bar is met without them.

### 3.3 Non-goals (Phase 3 explicitly will NOT do these)

- ❌ Computer Use any production flow. (Phase 6.)
- ❌ Feishu/Lark adapter. (Phase 4.)
- ❌ DingTalk adapter. (Phase 5.)
- ❌ Satori/Koishi or Vercel Chat SDK adapters. (Phase 7+.)
- ❌ Web console. (Phase 8.)
- ❌ Multi-runtime / SSH-tunneled remote codex. (P2 in
  `02-TECHNICAL-DECISIONS.md`.)
- ❌ Any public-facing WebSocket listener. (Project redline.)
- ❌ Any HTTP server in the daemon. (Project redline; no inbound
  network surface.)
- ❌ Codex CLI / TUI output parsing. (Project redline.)
- ❌ Replacing `Supervisor`'s pre-attached-broker invariant. (D16
  redline; production = Supervisor, runtime-send = dev/operator only.)
- ❌ Removing or weakening `runtime-send` CLI smoke. (Operator tool.)
- ❌ Replacing the closed `ChannelAdapter` interface. (D14 — capability
  matrix is the only escape; future adapters conform verbatim.)
- ❌ Replacing `bindActorPolicy` with first-actor-wins. (Phase 2
  redline.)
- ❌ "Trusting" Telegram-asserted user IDs without a SecurityPolicy

  `onPendingResolved`, `bindActorPolicy`, `resolve`, `enablePendingMode`,

  `TelegramShapeFakeChannelAdapter` constraints (callback_data ≤62B,
  60s deadline) translate cleanly to real Telegram. (Phase 2 inline-cited
  Bot API spec; risk is low.)
- A3 Mac mini host runs macOS with `launchd` available; `node --version`
  ≥ 24; `codex --version` ≥ 0.128.0 (post-upgrade baseline). Phase 3
  rebases onto the `chore/codex-upgrade-0.128` branch as part of T0.5.
- A4 SQLite via `better-sqlite3` (sync, WAL, embedded; no separate
  server) is the **preflight-required candidate** Phase 3 store
  (`02-TECHNICAL-DECISIONS.md` §10 pre-ratifies the SQLite choice).
  `kysely` for query building. Migrations as plain SQL files, not
  drizzle. **D39 (v2)** REQUIRES a load-time preflight before any
  storage implementation work begins; if `better-sqlite3` cannot
  install/load on the target Node 24 / macOS, T1 STOPS and the user
  decides between (a) Node 24's built-in `node:sqlite` (release
  candidate / active development per Node docs — proven by preflight
  only) or (b) abort the storage sliver and re-scope. There is **no
  pure-JS fallback** — v1's claim that `sqlite3` is pure-JS was
  incorrect (it has native bindings; just slower than `better-sqlite3`).
- A5 `grammY` is the right Telegram library. (`06-IM-ADAPTERS.md` §3
  pre-ratifies; alternatives are native Bot API + ad-hoc HTTP — more
  code to maintain, no upside.)
- A6 The `phase-2-approval-im-surface-complete` tag's commit will not
  shift. If Codex backfill surfaces issues that require new commits,
  those land on a `phase-2-review-nits` branch / new tag, not by
  rewriting the existing tag.
- A7 No team of multiple Phase 3 implementers — single-implementer
  TDD pace, one task at a time, mid-phase code review checkpoints.

- ❌ **No Telegram SDK / raw Update types / `bot_token` literal strings
  outside `packages/im-telegram/src/`** (v2: narrowed from v1's
  over-broad "no Telegram code anywhere"). Specifically: no `import`
  of `grammy`, no `InlineKeyboardButton`/`Update` types, no `bot_token`
  literal, no `parse_mode` literal outside that package. Config files
  may name adapter sections (`[adapters.telegram]`) and env-var names
  (`IM_TELEGRAM_BOT_TOKEN`); docs may reference the platform by name.
  Boundary test enforces.
- ❌ **No raw inbound-action handler in the daemon**. Daemon
  subscribes to `adapter.onAction`; never reads platform-specific
  raw payloads.
- ❌ **No `bindActorPolicy` call without a prior `SecurityPolicy.checkApprovalDestination`
  pass**. The daemon wire-up MUST gate `bindActorPolicy` behind the
  policy. Test asserts that an unauthorized chat never observes a
  card with actionable buttons.
- ❌ **No `adapter.sendCard` call without a prior `bindActorPolicy` call**
  (v2 D33 redline). The daemon wire-up MUST issue the callback token
  and bind the actor policy BEFORE sending the card to the IM platform.
  This closes the click-before-bind race v1 left open. Test using a
  mock adapter that throws if `sendCard` is invoked without a matching
  bound policy.
- ❌ **No `broker.resolve` call without callback-token-record
  validation** (v2 D34 / D35 redline). Daemon's inbound-action handler
  MUST: (1) look up the SQLite token record, (2) validate `record.messageRef
  === inbound.messageRef`, (3) atomically transition `status: bound →
  used`, (4) only THEN call `broker.resolve`. Stale token, missing
  messageRef, or already-used record fails closed (`answerAction(ok:false)`,
  no `resolve()` invocation, no audit leak).
- ❌ **No `broker.resolve` call without round-tripped
  `callbackNonce`**. Already enforced by `ResolveApprovalInput` shape
  (D19), but Phase 3 docs the call-site invariant.
- ❌ **Wrong actor / stale callback / duplicate decision / expired
  approval / transport-lost approval all fail closed at the broker
  boundary**. Phase 2 broker already does this; Phase 3 adds tests
  that exercise the path through the production daemon wire-up.
- ❌ **No bot tokens / API keys / cookies in any committed file or any
  test fixture**. Tokens load from `process.env.IM_TELEGRAM_BOT_TOKEN`
  (or a configured env var name) only. Logging uses `core/redact`'s
  14 patterns (token regex among them) on every audit / pino log
  serializer.
- ❌ **No public listener**. Daemon does not bind to any TCP/UDP port
  (`grammY`'s long-poll is OUTBOUND only; webhooks are off — no
  reverse-tunnel webhook config in Phase 3). Boundary test asserts
  no `net.createServer`/`http.createServer`/`net.Server` in production
  src.
- ❌ **No `/cu` command parsing** in Phase 3 CommandRouter (so a typo
  doesn't accidentally activate a Phase 6 surface that doesn't exist).
  CommandRouter explicitly errors with "Computer Use is not supported
  in this Phase" for any `/cu` / `/computer-use` prefix.
- ❌ **No DB migration script that auto-runs in production without
  the `--auto-migrate` config flag**. Default = false; daemon refuses
  to start if DB schema mismatches and `--auto-migrate` is unset.
- ❌ **No tokens or unredacted approval params in SQLite columns**.
  Audit table stores redacted JSON (already true in `core/audit`'s
  ring buffer; mirror to SQLite via the same redact path).
- ❌ **No `process.exit` from any production module other than the
  CLI/daemon entrypoints**. Supervisor halts via fatal audit; library
  code never exits.

## 7. Architecture decisions D22+

(Phase 1 D5–D10 + Phase 2 D11–D21 carry forward unchanged.)

- **D22 (v2 clarified)** — `SecurityPolicy` is a synchronous
  `check(input) → Decision` function injected into the daemon wire-up.
  Synchronous because every branch can be answered from in-memory
  tables (allowlist arrays, deny-pattern regexes). Async I/O at
  decision time is forbidden. **Reload semantics (v2)**: SIGHUP /
  admin CLI builds a NEW immutable policy snapshot from disk;
  swap is atomic single-assignment (`this.#current = newPolicy`).
  Failed reload (config validation error) leaves old policy active;
  daemon logs the error and continues. Concurrent in-flight checks
  see either old or new policy, never partial.
- **D23 → superseded by D38 (v2)** — see D38 below.
- **D24** — `@codex-im/im-telegram` consumes only
  `@codex-im/channel-core` types + `grammY` + Node stdlib. Boundary
  test asserts no import of `@codex-im/core`, `@codex-im/codex-runtime`,
  `@codex-im/app-server-client`. (Mirrors F13 for channel-core.)
- **D25** — Daemon wire-up is a class (`Daemon`), not a free
  function. Constructor takes a `DaemonOptions` injection bag of
  factories (security policy, session router, channel adapter, broker,
  supervisor, runtime, audit). `start()` and `stop()` are the only
  public methods. Test rigs construct a `Daemon` with the
  `TelegramShapeFakeChannelAdapter` (no separate test rig file).
  `phase2-e2e-rig.ts` stays as historical reference; new tests use
  `Daemon` directly.
- **D26** — `CommandRouter` lives in `@codex-im/core`. Pure function:
  `routeInbound(text) → { kind: "command", name, args } | { kind:
  "prompt", text, attachments }`. CommandRouter is platform-neutral;
  Telegram adapter just passes the text through. Slash commands
  (`/help`, `/projects`, `/new`, `/use`, `/status`, `/stop`) defined
  here.
- **D27** — `@codex-im/storage-sqlite` is a new package. Public surface:
  - `openDatabase(path) → Database` (idempotent, runs migrations iff
    `auto_migrate` opt is set).
  - Repository classes: `BindingRepository`, `ApprovalRepository`,
    `AuditRepository`, `EventLogRepository`. Each is a thin wrapper
    over typed SQL. No ORM.
  - Boundary test asserts no import of `@codex-im/core`,
    `@codex-im/codex-runtime`, `@codex-im/app-server-client`,
    `@codex-im/channel-core`. Storage is below core; core consumes
    storage via dependency injection.
- **D28** — `@codex-im/config` is a new package (or a sub-module of
  daemon). Loads TOML from `~/.codex-im-bridge/config.toml` (path
  configurable via `CODEX_IM_CONFIG`); validates with `zod`; resolves
  env-var references (`${ENV.IM_TELEGRAM_BOT_TOKEN}`); never logs the
  resolved values. (07-SECURITY §8.)
- **D29 (v2 rewritten)** — `Daemon.start()` enforces a strict
  initialization order. v1's order had two correctness bugs (no
  `enablePendingMode`; subscribe-then-bind race) that v2 fixes:

  ```text
  1.  load + validate config (zod);
  2.  preflight SQLite (D39): require better-sqlite3 to install/load;
      open DB; run migrations iff `--auto-migrate` set;
  3.  construct ApprovalBroker(placeholderClient) + broker.attach()
      (Phase 2 D16 contract: broker must be pre-attached before
      handing to Supervisor);
  4.  ★ NEW (D32) ★ for each method in the IM-routable pending-mode
      registry: broker.enablePendingMode(method);
  5.  construct SecurityPolicy + SessionRouter (D38 sync write-through);
  6.  construct Supervisor with the pre-attached broker;
  7.  Supervisor.start() — codex spawns; Supervisor.#spawnFresh swaps
      the broker's client reference via broker.reattach(realClient);
  8.  construct ChannelAdapter (Telegram) but DO NOT call adapter.start()
      yet — subscriptions must be wired first;
  9.  ★ NEW (D33) ★ wire onPendingCreated → two-phase token flow:
        broker.onPendingCreated(snap) →
          policy.checkApprovalDestination(snap, target):
            deny  → daemon issues synthetic system actor + nonce,
                    binds policy, broker.resolve(decline) (D36);
            allow → daemon issues opaque callback token (one per
                    visible action), generates callbackNonce, calls
                    broker.bindActorPolicy(approvalId, {allowedActors,
                    target, callbackNonce}) FIRST, then
                    projectAsRichBlock + render with token in
                    callback_data, then adapter.sendCard, then
                    daemon attaches MessageRef to token records in
                    SQLite (status: issued → bound).
  10. wire onAction → token-record-validated resolve (D34/D35):
        adapter.onAction(action) →
          decode token from action.callback_data ("v1:" + token) →
          look up token record in SQLite by hash(token) →
            no record OR status ≠ bound OR expired
                  → answerAction(ok:false, "stale or unknown") + return;
          validate record.messageRef === action.messageRef
                  → mismatch → answerAction(ok:false, "stale message");
          atomic SQLite UPDATE: status bound → used (CAS by status)
                  → CAS failure (someone else already used) →
                    answerAction(ok:false, "already resolved");
          policy.checkUserAndChat(action.target, action.sender)
                  → deny → answerAction(ok:false, "unauthorized");
          broker.resolve({approvalId: record.approvalId, decision:
            record.action, actor: action-derived actor, target:
            record.target, callbackNonce: record.callbackNonce}) →
          on result.kind:
            ok    → answerAction(ok:true) + adapter.updateCard(MessageRef,
                    terminal card);
            error → answerAction(ok:false, error.kind humanized).
  11. wire onMessage → policy → CommandRouter → SessionRouter →
      CodexRuntime.startTurn (or steerTurn, or /stop).
  12. adapter.start() — last; only now can inbound IM events arrive.
  13. (D37) on SIGTERM/SIGINT: invoke Daemon.stop() shutdown
      sequence below.
  ```

- **D30 (v2 rewritten)** — Telegram callback_data encoding is **NOT**
  `${approvalId}|${kind}|${nonce}` (v1, rejected). Real codex
  RequestId = `string | number` (`packages/codex-protocol/src/generated/RequestId.ts`)
  with no length bound; arbitrary RequestIds could overflow Telegram's
  64B limit. v2 uses an opaque token mapped via SQLite (D34):

  ```text
  callback_data = "v1:" + opaqueToken
    where opaqueToken = base32(crypto.randomBytes(10))[:16]   // 16 chars
    total bytes = 3 ("v1:") + 16 = 19  ≤  64 (Telegram) — large headroom.
  ```

  The daemon never embeds approvalId, decision kind, or nonce in
  callback_data. The opaque token is the ONLY thing on the wire; the
  SQLite `callback_tokens` table maps `tokenHash` → `(approvalId,
  action, callbackNonce, target, actor, messageRef, status, expiresAt)`.

- **D31 (v2 clarified)** — Audit ring buffer in `@codex-im/core` is
  the primary audit surface; SQLite is a write-through copy via a
  daemon-level `audit.onEmit(record => sqlite.insertRedacted(record))`
  subscriber. Sync write (better-sqlite3); failure does NOT block the
  broker. v2 adds rate-limiting on failure logs:
  - First failure: emit `audit.sqlite_unavailable` to the in-memory
    ring (one-time marker visible to operators).
  - Subsequent failures within a 60s window: aggregated to a single
    counter `daemon.audit_sink_dropped_count`, exposed via
    `codex-im daemon status`. No per-event log spam.

- **D32 (v2 NEW)** — IM-routable pending-mode registry. The 8
  ServerRequest methods Phase 3 daemon enables for pending-mode are
  enumerated in a typed const, sourced from the Phase 2 broker's
  `enablePendingMode<M>` method type parameter (which itself is
  `M extends PendingableServerRequestMethod`). The registry lives in
  `@codex-im/daemon/src/pending-methods.ts` and is type-checked
  against the broker's accepted-methods union; if codex adds or
  removes a pendingable method in a future protocol version, the
  registry fails to compile. Phase 2 already enumerates 8 methods in
  `phase2-e2e-rig.ts:77-86` (`DEFAULT_PENDING_METHODS` const) — that
  list is the v2 source of truth for which methods Daemon.start step 4
  enables. **Hard gate**: if the registry cannot be derived from
  Phase 2 broker types at T14/T15 implementation time, T14/T15 STOPS;
  the user decides between (a) extend Phase 2 broker types to expose
  the union or (b) abort Phase 3 daemon implementation until the
  registry is verified. No raw method literals outside this registry
  + Phase 2's two approved homes (`approval-broker.ts`,
  `approval-request-kind.ts`).

- **D33 (v2 NEW)** — Two-phase callback token lifecycle.
  Replaces v1's racy `sendCard → bindActorPolicy` ordering.
  Lifecycle states (persisted in SQLite `callback_tokens.status`):

  ```text
  issued  ── token row created in SQLite; bindActorPolicy invoked;
             card NOT yet sent. Inbound action with this token →
             fail closed "binding_not_ready"
  bound   ── adapter.sendCard returned MessageRef; daemon attached
             MessageRef to the token row. Inbound action with this
             token → eligible for resolve (subject to messageRef
             match + actor + nonce checks).
  used    ── broker.resolve was called for this token. Subsequent
             clicks → fail closed "already_resolved".
  expired ── TTL passed; SecurityPolicy expire-sweep transitions
             status. Inbound → fail closed "expired".
  revoked ── card was updated/replaced (e.g. status flip on resolve);
             old token rows for this approval are revoked. Inbound →
             fail closed "stale_token".
  ```

  Daemon flow:

  ```text
  policy allow → daemon: BEGIN TRANSACTION
                            INSERT callback_tokens (status='issued')
                              for each visible action;
                            COMMIT;
                          broker.bindActorPolicy(approvalId, {
                            allowedActors, target, callbackNonce: token-bind-nonce
                          });
                          // tokens are committed AND bound BEFORE remote send
                 daemon: adapter.sendCard(target, card-with-tokens-in-callback-data)
                           → returns MessageRef
                 daemon: UPDATE callback_tokens SET messageRef = ?, status = 'bound'
                          WHERE approval_id = ? AND status = 'issued';
  ```

  No card is ever rendered with a token whose policy is not bound.
  No bind ever happens AFTER send. The race window in v1 is closed.

- **D34 (v2 NEW)** — `callback_tokens` repository in
  `@codex-im/storage-sqlite`. Schema:

  ```sql
  CREATE TABLE callback_tokens (
    token_hash       TEXT NOT NULL PRIMARY KEY,
    approval_id      TEXT NOT NULL,
    action           TEXT NOT NULL CHECK(action IN
                       ('allow_once','allow_session','decline','cancel')),
    callback_nonce   TEXT NOT NULL,
    target_key       TEXT NOT NULL,
    actor_kind       TEXT NOT NULL CHECK(actor_kind IN ('im','system')),
    actor_user_id    TEXT,
    actor_platform   TEXT,
    actor_reason     TEXT,
    msg_chat_id      TEXT,
    msg_message_id   TEXT,
    status           TEXT NOT NULL CHECK(status IN
                       ('issued','bound','used','expired','revoked')),
    created_at       TEXT NOT NULL,
    expires_at       TEXT NOT NULL
  );
  CREATE INDEX idx_callback_tokens_approval ON callback_tokens(approval_id);
  CREATE INDEX idx_callback_tokens_status_expires
    ON callback_tokens(status, expires_at);
  ```

  Token storage policy:

  ```text
  raw_token  = base32(crypto.randomBytes(10))[:16]  ← 16 chars, ≥80 bits
                                                     entropy. NEVER
                                                     stored.
  token_hash = sha256(raw_token).slice(0, 32)       ← stored as PRIMARY
                                                     KEY. SQLite leak
                                                     does not compromise
                                                     in-flight callbacks.
  callback_data on the wire = "v1:" + raw_token    ← only the IM
                                                     platform sees the
                                                     raw token, in the
                                                     callback only.
  ```

  Status transitions are atomic single-row UPDATEs with `WHERE status = ?`
  guards. Used-token replay (race between two simultaneous clicks)
  fails closed because the second UPDATE returns 0 rows changed.

- **D35 (v2 NEW)** — MessageRef validation before `broker.resolve`.
  `InboundAction` (`packages/channel-core/src/types.ts`) is extended
  to carry the platform's message identifier:

  ```ts
  export type InboundAction = {
    readonly approvalId: string;
    readonly uiAction: ApprovalAction;
    readonly target: Target;
    readonly sender: Sender;
    readonly callbackNonce: string;
    readonly receivedAt: Date;
    readonly callbackHandle: string;
    readonly messageRef: MessageRef;   // ★ NEW in v2
  };
  ```

  Daemon validates `record.messageRef === inboundAction.messageRef`
  (deep equal on `{chatId, messageId}`) BEFORE calling `broker.resolve`.
  Mismatch → `answerAction(ok:false, "stale message")` + audit
  `approval.stale_message_ref`. **The validation lives at the daemon
  callback-token-record level, NOT inside ActorPolicy** — extending
  `ActorPolicy.messageRef` was rejected because it would force binding
  to happen AFTER `sendCard` returns the MessageRef, reintroducing the
  D33 race. v2's design issues the token + binds the actor BEFORE
  sendCard, then attaches MessageRef to the token-record (not the
  ActorPolicy) once sendCard returns.

- **D36 (v2 NEW)** — Synthetic system actor for SecurityPolicy
  auto-decline. Verified during v2 design that
  `ApprovalActor = null | {kind:"system", reason: string} | {kind:"im", ...}`
  already exists in `packages/core/src/types.ts` and that
  `ResolveApprovalInput.actor: NonNullable<ApprovalActor>` and
  `ActorPolicy.allowedActors: readonly NonNullable<ApprovalActor>[]`
  both accept the system kind. **No broker code change is needed.**
  Auto-decline daemon flow:

  ```ts
  // SecurityPolicy.checkApprovalDestination returns
  //   {kind: "auto_decline", reason: "policy_denied"}
  const sysActor: ApprovalActor =
    {kind: "system", reason: "policy_denied"};
  const sysNonce = crypto.randomUUID();
  broker.bindActorPolicy(snap.approvalId, {
    allowedActors: [sysActor],
    target,
    callbackNonce: sysNonce,
  });
  await broker.resolve({
    approvalId: snap.approvalId,
    decision: {kind: "decline"},
    actor: sysActor,
    target,
    callbackNonce: sysNonce,
  });
  // audit: approval.policy_denied_auto_decline
  // adapter is never called; user never sees a card.
  ```

  No new broker bypass API. Settlement still goes through
  `broker.#settleEntry` (Phase 2 D21 byte-identical). The wire response
  to codex is the standard decline shape per kind (D11 mapper).

- **D37 (v2 NEW)** — Shutdown settlement order. `Daemon.stop()` runs
  in this fixed sequence:

  ```text
  1. adapter.pauseInbound() — stop accepting new IM events;
     in-flight onAction handlers continue.
  2. broker.failPendingAsTransportLost("daemon_shutdown") — every
     pending entry transitions to transport_lost terminal; wire
     responses (decline) go out before client.stop.
  3. await microtask drain (small fixed delay, e.g. 100ms) so
     in-flight wire responses land before transport closes.
  4. supervisor.stop() — codex subprocess clean exit.
  5. adapter.stop() — close long-poll, drop subscribers.
  6. storage.close() — SQLite WAL checkpoint.
  ```

  Order matters: step 2 before step 4 means the codex subprocess
  receives the decline responses before the transport closes; step 5
  before step 6 ensures no SQLite writes happen after close.

- **D38 (v2 NEW, supersedes D23)** — `SessionRouter` synchronous
  write-through. v1's "async write-through" conflicted with G3
  (binding restart-survival). v2 contract:

  ```text
  router.bind(target, projectId)   ← synchronous SQLite UPSERT
                                     before return; on write failure,
                                     throws; in-memory cache NOT
                                     updated.
  router.bindThread(target, threadId) ← same sync pattern.
  router.resolve(target)           ← in-memory cache hit (fast);
                                     cache miss → sync SQLite read,
                                     populate cache, return.
  ```

  `/use`, `/new` slash commands MUST `await router.bind()` (or
  `bindThread`) before sending the user-facing acknowledgement
  message. SQLite write failure → command fails with operator-readable
  error; binding NOT silently lost.

- **D39 (v2 NEW)** — SQLite native preflight + fallback policy.
  T1 begins with:

  ```bash
  pnpm add better-sqlite3 kysely
  node -e 'const db = require("better-sqlite3")(":memory:"); \
    db.prepare("select 1 as x").get(); console.log("better-sqlite3 ok");'
  ```

  If preflight passes: proceed with `better-sqlite3`. Storage tasks
  unblocked.

  If preflight fails: STOP. Storage tasks blocked. The user decides
  between two paths:
  - (a) Try Node 24's built-in `node:sqlite` as a fallback CANDIDATE,
    AFTER a separate preflight that proves both (i) the sync API
    needed by D34/D38 is supported on the running Node version and
    (ii) all storage-sqlite tests pass against `node:sqlite`. **Note**
    per Node docs `node:sqlite` is currently release-candidate /
    active-development; treat as preflight-required, not a free
    swap-in.
  - (b) Abort the storage sliver; re-scope Phase 3 to defer SessionRouter
    persistence to Phase 4 (G3 acceptance criterion would not be met).

  v2 explicitly removes v1's claim that `sqlite3` (npm) is a "pure-JS
  fallback" — `sqlite3` has native bindings, just slower than
  `better-sqlite3`. Not a real fallback for the build-failure case.

#### `callback_tokens` (v2 / D34) — full schema

```sql
CREATE TABLE callback_tokens (
  token_hash       TEXT NOT NULL PRIMARY KEY,
  approval_id      TEXT NOT NULL,
  action           TEXT NOT NULL CHECK(action IN
                     ('allow_once','allow_session','decline','cancel')),
  callback_nonce   TEXT NOT NULL,

  actor_kind       TEXT NOT NULL CHECK(actor_kind IN ('im','system')),
  actor_user_id    TEXT,
  actor_platform   TEXT,
  actor_reason     TEXT,

  expires_at       TEXT NOT NULL
);
CREATE INDEX idx_callback_tokens_approval
  ON callback_tokens(approval_id);
CREATE INDEX idx_callback_tokens_status_expires
  ON callback_tokens(status, expires_at);
```

Hashing policy: store `sha256(rawToken).slice(0, 32)` only. Raw token
is held in process memory just long enough to render `callback_data`
on the wire; never persisted, never logged. SQLite leak does not
compromise live tokens.

**Population of `actor_*` columns** (★ v2.1 P2-A clarification ★).
The schema requires `actor_kind` NOT NULL but the §10.2 D33 issue
flow can't yet know the user's identity at INSERT time (the policy
issues a token allowed-actor-list before any human has clicked).
v2.1 contract:

- At INSERT (D33 step 1): `actor_kind` is set per the policy that's
  about to be bound — `'im'` for normal authorized flow (§10.2
  policy-allow branch); `'system'` for D36 auto-decline. `actor_user_id`
  / `actor_platform` columns stay NULL at INSERT.
- At click-time CAS (§10.3 step 3): `actor_user_id` and
  `actor_platform` are populated atomically with the
  `bound → used` transition, set from the inbound action's
  authenticated platform identity. `actor_kind` does not change.
- Schema integrity: NOT NULL holds at all times because `actor_kind`
  is set at INSERT; only `actor_user_id` and `actor_platform`
  transition NULL→VALUE.

Status state machine: see D33. Atomic transitions via single-row
UPDATE with `WHERE status = ?` guards (CAS).

Expiry: `expires_at` matches `approvalTtlMs` (default 30 min). Lazy
expire on lookup AND a periodic prune (G9). Used/revoked rows kept
≤ 7 days for audit, then pruned.

## 10. Security model

### 10.1 Inbound message gating

```text
Telegram update
  → adapter.onMessage handler (in @codex-im/im-telegram)
  → Daemon.handleInboundMessage(msg)
    → SecurityPolicy.checkUserAndChat(target, sender)
      ├── deny? → drop silently (no reply, no audit beyond a
      │           "denied unauthorized inbound" record)
      └── allow → CommandRouter.route(msg.text)
            ├── command? → Daemon.handleCommand(...)
            └── prompt?  → SessionRouter.resolve(target)
                  → CodexRuntime.startTurn(...)
```

### 10.2 Approval gating (v2 — D33 two-phase token + D36 system actor)

```text
broker.onPendingCreated(snap)
  → SecurityPolicy.checkApprovalDestination(snap, target)
    ├── auto_decline (D36 synthetic system actor — no card rendered)
    │     daemon: const sysActor = {kind:"system", reason:"policy_denied"};
    │             const sysNonce = crypto.randomUUID();
    │             broker.bindActorPolicy(snap.approvalId, {
    │               allowedActors: [sysActor], target, callbackNonce: sysNonce
    │             });
    │             await broker.resolve({
    │               approvalId, decision: {kind:"decline"},
    │               actor: sysActor, target, callbackNonce: sysNonce
    │             });
    │             audit: approval.policy_denied_auto_decline
    │             // codex receives standard decline shape (D11 mapper).
    │             // user never sees a card.
    │
    └── allow → daemon (D33 two-phase token flow):
          1. for each visible action in card.actions:
               raw = base32(crypto.randomBytes(10))[:16]
               record: {token_hash: sha256(raw)[:32], approval_id,
                        action, callback_nonce, target, actor: null
                        (filled at click time), msg_*: null, status:'issued',
                        created_at, expires_at}
               INSERT into callback_tokens (single TX);
          2. broker.bindActorPolicy(snap.approvalId, {
               allowedActors: [{kind:"im", platform, userId: <chat allowlist>}],
               target,
               callbackNonce: <single bind-nonce per approval>,
             });
             // bind happens BEFORE remote send (closes v1 race);
             // bind-nonce is independent from per-token raw bytes;
             // tokens point to records, broker validates the bind-nonce.
          3. block = projectAsRichBlock(snap)
             card  = block.card with each action's callback_data
                     replaced by "v1:" + rawToken  (raw, not hash)
          4. result = await adapter.sendCard(target, card);
             // returns {messageRef, callbackNonce: telegram-nonce}
          5. UPDATE callback_tokens SET msg_chat_id=?, msg_message_id=?,
                                          status='bound'
                                  WHERE approval_id=? AND status='issued';
             // multi-row UPDATE; all action tokens flip to 'bound'
             // atomically. Inbound action arriving BEFORE this update →
             // sees status='issued' → fails closed "binding_not_ready".

  failure modes:
    - SQLite INSERT fails → daemon rolls back; broker entry stays
      pending; daemon emits audit.token_persist_failed; renderer-defensive
      decline-only fallback may apply.
    - bindActorPolicy fails (e.g. duplicate) → daemon rolls back tokens;
      fail closed.
    - adapter.sendCard fails (e.g. callback_data overflow — impossible by
      D30 construction, but defensive) → daemon rolls back tokens; broker
      entry transitions to expired via TTL; no user-visible card.
    - ★ v2.1 (gstack round-2 P1-A) ★
      Step-5 UPDATE issued→bound fails AFTER adapter.sendCard already
      returned (i.e. card is already on the user's screen). Without
      this branch the v2 design leaves tokens stuck at 'issued'
      forever; clicks fail closed "binding_not_ready"; only TTL
      eventually frees the broker entry. v2.1 contract:
        a. retry the UPDATE up to 3 times with linear backoff (50ms
           / 150ms / 350ms);
        b. on persistent failure: emit `audit.token_bind_persist_failed`
           with approvalId + tokenHash count; mark the broker entry
           for early sweep by setting an in-memory flag the G9 sweep
           checks (see T19e.1) — sweep transitions any 'issued'
           token older than 5s for an approval with this flag to
           'revoked' status, freeing the broker entry via
           broker.failPendingAsTransportLost equivalent for that
           single approvalId (uses broker's existing settle path);
        c. user-visible: card stays on screen but every click fails
           closed "binding not ready"; once sweep runs, broker
           records the approval as transport_lost and the card's
           updateCard happens on the next IM event for that target
           (or stays stale until card retention TTL — acceptable
           since the codex side has been declined).
      Test: P3.T-Sec-step5-failure (new) injects SQLite write error
      after sendCard succeeds; asserts retry sequence + audit emit +
      sweep-driven broker settle.
```

`SecurityPolicy.checkApprovalDestination` returns `{kind: "allow"} |
{kind: "auto_decline", reason: string}`. The auto-decline path
records an audit event with the reason and never reaches the IM
layer (no card sent, no token issued).

### 10.3 Action gating (v2 — D34 token lookup + D35 messageRef validation)

```text
adapter.onAction(action)
  → Daemon.handleInboundAction(action)
    → 0. decode rawToken from action.callback_data ("v1:" prefix);
       no prefix or wrong format → answerAction(ok:false, "stale or unknown")
                                  + audit approval.malformed_callback_data
                                  + return.
    → 1. record = callbackTokens.findByHash(sha256(rawToken)[:32])
         no record               → answerAction(ok:false, "stale or unknown");
         status === 'expired'    → answerAction(ok:false, "expired");
         status === 'revoked'    → answerAction(ok:false, "stale token");
         status === 'used'       → answerAction(ok:false, "already resolved");
         status === 'issued'     → answerAction(ok:false, "binding not ready")
                                  + audit approval.callback_before_bind
                                  + return.
         (only status === 'bound' continues)
    → 2. messageRef validation (D35):
         (★ v2.1 P1-B ★) Adapter-level guard: if the IM platform's
         underlying message field is null (Telegram callback_query.message
         can be null for >48h old messages, inline-mode results, or
         user-deleted messages — Bot API Optional), the adapter MUST
         set inboundAction.messageRef.messageId = "<unknown>". Daemon
         step 2 then fails closed:
            messageId === "<unknown>" → answerAction(ok:false,
                                          "stale message (cannot validate)")
                                       + audit approval.message_ref_unknown
                                       + return;
         Otherwise (normal path):
         deepEqual(record.{msg_chat_id, msg_message_id},
                   action.messageRef.{chatId, messageId}) ?
            no → answerAction(ok:false, "stale message")
               + audit approval.stale_message_ref + return.
    → 3. atomic CAS: UPDATE callback_tokens
                       SET status='used', actor_user_id=?, actor_platform=?
                     WHERE token_hash=? AND status='bound';
         rowsAffected === 0 → another click won the race;
                              answerAction(ok:false, "already resolved")
                              + return.
         (rowsAffected === 1 means we own the resolve)
    → 4. SecurityPolicy.checkUserAndChat(action.target, action.sender):
         deny → answerAction(ok:false, "unauthorized")
              + audit approval.policy_denied_inbound + return.
    → 5. result = await broker.resolve({
           approvalId: record.approval_id,
           decision: record.action,           // the action this token was for
           actor: {kind:"im", platform: action.target.platform,
                   userId: action.sender.userId},
           target: record.target,
           callbackNonce: record.callback_nonce,
         });
    → 6. on result:
         ok           → answerAction(ok:true, "decision recorded")
                      + adapter.updateCard(record.messageRef, terminal-card)
                      + UPDATE callback_tokens SET status='revoked'
                          WHERE approval_id = ? AND status = 'bound';
                          // sibling tokens (other actions) revoked.
         error.kind:
           wrong_actor       → answerAction(ok:false, "wrong actor");
           stale_callback    → answerAction(ok:false, "stale nonce");
           wrong_target      → answerAction(ok:false, "wrong target");
           binding_required  → answerAction(ok:false, "internal: missing bind")
                              + audit approval.binding_required_unexpected
                              (should be unreachable post-D33; defense in
                              depth).
           expired           → answerAction(ok:false, "expired");
           transport_lost    → answerAction(ok:false, "codex restarted, retry");
           duplicate_decision → answerAction(ok:false, "already resolved");
           unsupported_decision → answerAction(ok:false, "invalid action");
           unknown_approval_id → answerAction(ok:false, "approval not found")
                                + audit approval.unknown_approval_id_after_bound
                                (defense in depth — if status='bound' but
                                broker says unknown_approval_id, daemon
                                got out of sync. Likely cause: daemon
                                restart with broker reset but SQLite
                                token survived.)
```

The CAS step (3) protects against duplicate clicks racing. The
messageRef validation step (2) closes the stale-card-after-edit
attack. Steps 1+2+3 ALL run BEFORE `broker.resolve`; broker only
sees well-formed, single-resolution attempts that have passed
adapter-level platform-asserted checks.

### 10.5 Token handling + shutdown (v2 — D37 settlement order)

**Token handling**:

- Bot token sourced from `process.env[<envVarName>]` at daemon start.
  If absent, daemon refuses to start with a clear error (no Telegram
  adapter constructed, daemon halts).
- Token never stored in SQLite, never written to logs, never sent
  through any audit event, **never rendered into the launchd plist**
  (D-Op-1). `redact` patterns include the Telegram bot-token shape
  (`\d+:[A-Za-z0-9_-]{35}`).
- Test asserts that even an intentional `audit.emit({metadata: {token:
  '1234567890:AAaaaa…'}})` is redacted in the ring buffer + the SQLite
  write-through copy.
- launchd plist references env-var name only; daemon resolves token
  from environment (or macOS Keychain via launchd's environment
  loader) at runtime. The plist itself contains no secret material.

**Shutdown sequence (D37)**:

```text
SIGTERM / SIGINT received  (or operator: codex-im daemon stop)
   → Daemon.stop()
        1. adapter.pauseInbound()         // stop new inbound, keep
                                           // in-flight handlers running
        2. broker.failPendingAsTransportLost("daemon_shutdown")
                                           // every pending broker entry
                                           // → transport_lost terminal;
                                           // wire decline goes out before
                                           // codex transport closes.
        3. await drain (small fixed delay, ~100ms or actual flush)
                                           // in-flight wire responses
                                           // land before close.
        4. supervisor.stop()              // codex subprocess clean exit
        5. adapter.stop()                 // close long-poll / drop subs
        6. storage.close()                // SQLite WAL checkpoint
   → process exits 0
```

Order is load-bearing: step 2 BEFORE step 4 means codex sees the
decline before the wire closes; step 5 BEFORE step 6 means no SQLite
writes after close. SIGTERM during shutdown (impatient operator)
halts the current step but does NOT skip step 2 — `Daemon.stop` is
idempotent and re-entrant.

## 13. Operational strategy

### 13.1 launchd (v2 — D-Op-1: no token rendered into plist)

**Redline (D-Op-1)**: the installed plist contains NO secret material.
Bot token is loaded from environment (or macOS Keychain via a wrapper
script) at daemon runtime, not embedded in the plist file. The plist
only references the env-var NAME — the value is set by either (a)
launchd loading from the user's environment at agent load time or
(b) a wrapper shell script that pulls from Keychain via `security
find-generic-password` and `exec`s the node binary.

**Templated source** (`templates/io.codex-im-bridge.plist.tmpl`):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
  <dict>
    <key>Label</key>
    <string>io.codex-im-bridge</string>
    <key>ProgramArguments</key>
    <array>
      <string>{{NODE_BIN}}</string>
      <string>{{HOME}}/.codex-im-bridge/bin/daemon.mjs</string>
    </array>
    <key>RunAtLoad</key><true/>
    <key>KeepAlive</key>
    <dict>
      <key>SuccessfulExit</key><false/>
      <key>Crashed</key><true/>
    </dict>
    <key>StandardOutPath</key>
    <string>{{HOME}}/.codex-im-bridge/logs/daemon.log</string>
    <key>StandardErrorPath</key>
    <string>{{HOME}}/.codex-im-bridge/logs/daemon.err.log</string>
    <!--
      Token loading strategy: NO token value is rendered into the plist.
      Choice 1 (recommended): wrap with a Keychain-loading shell script
        that exec's node:
            ProgramArguments → ["{{HOME}}/.codex-im-bridge/bin/load-and-run.sh"]
        and the script does:
            export IM_TELEGRAM_BOT_TOKEN="$(security find-generic-password \
              -s codex-im-bridge -a $USER -w)"
            exec {{NODE_BIN}} {{HOME}}/.codex-im-bridge/bin/daemon.mjs
      Choice 2 (less secure but simpler): launchctl setenv
        IM_TELEGRAM_BOT_TOKEN <value>  (one-time, by operator) — the
        env-var lives in the launchd session, NOT the plist file.
      The installer asks the operator which choice and acts accordingly;
      neither writes the token into the plist.
    -->
  </dict>
</plist>
```

**Installer** (`bin/install-launchd.mjs`):
- Reads `$HOME` and `$USER` from environment.
- Resolves `{{NODE_BIN}}` via `process.execPath` (the running node) or
  `which node`.
- Replaces `{{NODE_BIN}}` and `{{HOME}}` placeholders.
- Writes the rendered plist to `$HOME/Library/LaunchAgents/io.codex-im-bridge.plist`.
- Optionally writes the load-and-run.sh wrapper if Choice 1.
- Runs `launchctl load $HOME/Library/LaunchAgents/io.codex-im-bridge.plist`.

**Uninstaller** (`bin/uninstall-launchd.mjs`):
- `launchctl unload $HOME/Library/LaunchAgents/io.codex-im-bridge.plist`.
- Removes the plist file.
- Does NOT touch Keychain or env-var settings (operator may still want
  the token).

The installed plist always contains placeholders RESOLVED to the
operator's actual `$HOME` and node binary — no `/Users/mini` or
`/usr/local/bin/node` literals. This makes the plan portable across
Mac mini hosts and across operators.

### 13.2 Log rotation

`pino` with `pino-roll` rotation in `daemon.log`. Daily rotation, keep
14 days. No rotation in test/dev.

### 13.3 Observability minimum

- Log lines with `pid`, `name`, `level`, `time` (already in Phase 1
  pino setup).
- `codex-im daemon status` CLI lists: pid, uptime, current Codex thread
  count, pending approval count, last codex spawn time, supervisor
  failure count, last fatal (if any).
- No metrics endpoint, no Prometheus, no public observability surface
  in Phase 3.

### 13.4 Backups

Per `08-DATA-MODEL.md` §7: nightly copy of `state.db` →
`~/.codex-im-bridge/backups/state-YYYYMMDD.db`, keep 30. Phase 3 ships
the `pnpm run db:backup` script + a `cron` example; auto-scheduling
via launchd is a P1 nice-to-have (one extra plist).

### 13.5 Failure handling

- Daemon refuses to start on missing bot token / config / SQLite path
  permission errors. Exits non-zero with a clear message.
- Codex CLI hung on the production host: `Supervisor` halts after 5
  consecutive failures (Phase 1 D16); fatal logs surface to launchd's
  `StandardErrorPath`.
- SQLite write failure (disk full, etc.): daemon logs ERROR and
  continues with in-memory only. After-action: surface via
  `daemon status`.

## 20. Rollback strategy

Phase 3 lands on `phase-3-implementation` branch (separate from this
`phase-3-planning` branch). Each task lands as a separate commit; tag
candidates only apply at the end.

If a regression is found mid-phase:

- **Per-task**: revert the specific commit. Phase 1/2 commits stay
  intact. Re-do.
- **Phase-wide**: if a structural decision (e.g. D29 init order) is
  wrong, branch off `phase-2-approval-im-surface-complete` again,
  cherry-pick the still-good commits, drop the bad ones, re-run all
  reviews.
- **Production launchd uninstall**: `pnpm run launchd:uninstall`
  removes the plist; daemon stops; no other system state changes
  (no daemon-managed system files outside `~/.codex-im-bridge/`).
- **DB migration rollback**: each migration has a `-- DOWN:` comment
  with the SQL to undo it. `pnpm run db:rollback <version>` runs them
  in reverse. Phase 3 ships the runner; running it manually is the
  rollback procedure.
- **Telegram bot lockout**: revoke bot token via @BotFather; rotate
  via `IM_TELEGRAM_BOT_TOKEN` env var; daemon restart picks up the new
  token.

A force-rollback to "no Phase 3" is `git checkout
phase-2-approval-im-surface-complete` + `pnpm run launchd:uninstall`
+ remove SQLite. The Phase 2 tag's surface is untouched; only the
operational surface needs cleanup.

  - Mitigation: T20–T28 split into single-test sub-tasks; raw-update

- **R5 (rewritten v2)** — SQLite native-build risk + fallback policy.
  `better-sqlite3` requires native compilation against the host
  Node 24 toolchain. Failure modes: missing build tools, Node ABI
  mismatch, macOS Apple Silicon vs Intel binary mismatch.
  - Severity: MEDIUM.
  - Mitigation: D39 mandates a load-time preflight at T1.0. If it
    fails: STOP (do not proceed with storage tasks). The user decides:
    - (a) Try Node 24's built-in `node:sqlite` as a fallback CANDIDATE,
      AFTER a separate preflight that proves (i) the sync API our
      code needs is supported on the host's Node version (per Node
      docs `node:sqlite` is currently release-candidate / active-
      development; v1's "no-risk fallback" claim was wrong) and
      (ii) all storage-sqlite tests pass against `node:sqlite`.
    - (b) Abort the storage sliver and re-scope Phase 3 to defer
      SessionRouter persistence to Phase 4 (G3 acceptance not met).
  - **v1's `sqlite3` (npm) "pure-JS fallback" claim is removed** —
    `sqlite3` has native bindings, just slower than `better-sqlite3`.
    Not a real fallback for the build-failure case.

- **R6 (clarified v2)** — `pnpm protocol:check` fails on
  `phase-3-planning` branch (CODEX_VERSION=0.125 vs local codex
  0.128). NOT a v2 plan defect; resolved by T0.5 rebase onto
  `chore/codex-upgrade-0.128` (`d999af5`). Recorded here for
  auditability.

- **R7** — launchd installation requires user-specific paths; cross-
  user automation is fragile.
  - Severity: LOW (unchanged).
  - Mitigation v2: `bin/install-launchd.mjs` reads `$HOME` + `$USER`
