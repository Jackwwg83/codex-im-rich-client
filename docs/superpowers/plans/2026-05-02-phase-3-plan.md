## 1. Phase 3 mission

**Telegram MVP, end-to-end, production-shaped**: ship the smallest
slice that lets a real user issue a prompt over Telegram, watch
streaming output, click an approval button, and have the daemon stay
up across codex restarts and Mac mini reboots — without any test rig
in the call path.

Concretely, that bundles five interdependent slivers (deliberately
narrow scopes; nothing in this list pulls in CU, Lark, DingTalk, or
Web Console):

1. **Production daemon wire-up** (`@codex-im/daemon`) — replace the
   test-only `phase2-e2e-rig.ts` daemon-wireup function with a
   production module that subscribes
   `broker.onPendingCreated → projectAsRichBlock → adapter.sendCard
    → bindActorPolicy` and `adapter.onAction → SecurityPolicy.check
    → broker.resolve → adapter.answerAction → adapter.updateCard`,
   plus a CommandRouter and a SessionRouter (chat ⇄ project ⇄ thread).
2. **SecurityPolicy ACL** (`@codex-im/core`) — Phase 2 left
   `SecurityPolicy` as a `phase1-noop` mention in `types.ts`. Phase 3
   ships a real `SecurityPolicy` that runs BEFORE `bindActorPolicy`
   (so unauthorized chats never see the buttons) AND runs again
   inside the broker → resolve path (defense in depth). Includes
   user/chat/project allowlists, command deny patterns, and the
   "unknown method ⇒ critical, decline-only" defensive renderer
   path that's already partially implemented.
3. **`@codex-im/im-telegram`** (new package) — real Telegram
   ChannelAdapter built on `grammY`, conforming to the closed
   `ChannelAdapter` interface verbatim. Mirrors the constraints
   `TelegramShapeFakeChannelAdapter` already enforces (callback_data
   ≤62B, 60s answer-callback-query deadline, no parse_mode reliance).
4. **Persistent SessionRouter** (`@codex-im/core` + new
   `@codex-im/storage-sqlite`) — chat→project→thread bindings stored
   in SQLite so the daemon can survive a restart with bindings intact
   (a P0 acceptance criterion in 09-ROADMAP §Phase 2).
5. **launchd integration** (operational, in `packages/daemon` or a
   dedicated `packages/launchd`) — installable plist + log rotation
   wiring so the Mac mini "always-on daemon" promise is actually
   keepable.

**Why bundle them.** Each one in isolation is a half-feature: a
daemon wire-up with no Telegram has nothing to wire up; Telegram
without SecurityPolicy is a broken-by-design open relay; Telegram
with no SQLite means every codex restart loses bindings; without
launchd none of it survives a reboot. The bundle is the smallest
slice that lets Phase 4 (Lark) start cleanly.

**What is explicitly NOT in this mission**: see §3 Non-goals.

## 3. Goals and Non-goals

### 3.1 Goals (P0 — must ship)

- G1 **End-to-end Telegram-driven turn**: user sends text in a Telegram
  chat the daemon is configured to listen on; daemon starts a Codex
  thread (or reuses the bound thread); streams agent output back; if
  the model requests command execution / file change / permissions,
  daemon renders the approval card with inline keyboard; user clicks
  allow/decline; broker resolves; codex sees the decision; turn
  completes; daemon edits the streaming message with the final
  outcome.
- G2 **SecurityPolicy fail-closed gating**: an unauthorized
  user/chat/project combination cannot trigger a turn (inbound message
  dropped) AND cannot resolve an approval (broker's `bindActorPolicy`
  is never called for unauthorized targets, so any inbound action
  hits `binding_required`).
- G3 **Daemon restart preserves bindings**: chat→project→thread
  bindings live in SQLite (WAL); after `pkill -SIGTERM daemon` the
  next start picks up where it left off.
- G4 **launchd-installable**: a `bin/` script (or `pnpm run
  launchd:install`) installs a plist; `pnpm run launchd:uninstall`
  removes it; logs rotate; restart-on-crash works.
- G5 **No regression on Phase 2 internal gates**:
  typecheck/test/lint/method-literal-boundary/F13 channel-core
  boundary all stay green.
- G6 **Telegram callback_data within real platform constraints**:
  encoding works under Telegram Bot API limits (≤64B; we already
  enforce ≤62B in the fake for headroom).
- G7 **Token redaction**: bot token never appears in any audit event
  payload, log line, or thrown error message.

### 3.2 Goals (P1 — ship if cheap, defer otherwise)

- G8 Synthesized `turn_failed` events on transport-loss (one-line
  Phase 2 supervisor TODO).
- G9 Lazy-prune sweep for terminal `#pendingById` records.
- G10 `/status`, `/projects`, `/use`, `/new`, `/stop` slash commands
  for the Telegram chat (the minimum command set listed in 01-PRD §6).
  G1 only requires inbound prompt → outbound reply; the slash commands
  are operationally useful but the e2e bar is met without them.

### 3.3 Non-goals (Phase 3 explicitly will NOT do these)

- ❌ Computer Use any production flow. (Phase 6.)
- ❌ Feishu/Lark adapter. (Phase 4.)
- ❌ DingTalk adapter. (Phase 5.)
- ❌ Satori/Koishi or Vercel Chat SDK adapters. (Phase 7+.)
- ❌ Web console. (Phase 8.)
- ❌ Multi-runtime / SSH-tunneled remote codex. (P2 in
  `02-TECHNICAL-DECISIONS.md`.)
- ❌ Any public-facing WebSocket listener. (Project redline.)
- ❌ Any HTTP server in the daemon. (Project redline; no inbound
  network surface.)
- ❌ Codex CLI / TUI output parsing. (Project redline.)
- ❌ Replacing `Supervisor`'s pre-attached-broker invariant. (D16
  redline; production = Supervisor, runtime-send = dev/operator only.)
- ❌ Removing or weakening `runtime-send` CLI smoke. (Operator tool.)
- ❌ Replacing the closed `ChannelAdapter` interface. (D14 — capability

- ❌ Replacing `bindActorPolicy` with first-actor-wins. (Phase 2

- ❌ "Trusting" Telegram-asserted user IDs without a SecurityPolicy

## 6. Phase 3 redlines (carry forward to Phase 4+)

In addition to the project-wide redlines (CLAUDE.md) and the Phase 2
redlines (handoff §"Phase 2 redlines"):

- ❌ **No raw `AppServerClient.request("...")` from `@codex-im/im-telegram`,
  `@codex-im/daemon`, `@codex-im/storage-sqlite`, or any future
  `packages/im-*`**. Daemon wire-up uses `CodexRuntime` wrappers; IM
  adapters use only `ChannelAdapter`. Method-literal boundary test
  (T20) coverage extends to these new packages.
- ❌ **No Telegram-specific code outside `packages/im-telegram/src/`**.
  No `Bot`, `Update`, `InlineKeyboardButton`, `bot_token`,
  `parse_mode`, `chat_id` types or strings outside that package.
  (Boundary test enforces.)
- ❌ **No raw inbound-action handler in the daemon**. Daemon
  subscribes to `adapter.onAction`; never reads platform-specific
  raw payloads.
- ❌ **No `bindActorPolicy` call without a prior `SecurityPolicy.check`
  pass**. The daemon wire-up MUST gate `bindActorPolicy` behind the
  policy. Test asserts that an unauthorized chat never observes a
  card with actionable buttons.
- ❌ **No `broker.resolve` call without round-tripped
  `callbackNonce`**. Already enforced by `ResolveApprovalInput` shape
  (D19), but Phase 3 docs the call-site invariant.
- ❌ **Wrong actor / stale callback / duplicate decision / expired
  approval / transport-lost approval all fail closed at the broker
  boundary**. Phase 2 broker already does this; Phase 3 adds tests
  that exercise the path through the production daemon wire-up.
- ❌ **No bot tokens / API keys / cookies in any committed file or any
  test fixture**. Tokens load from `process.env.IM_TELEGRAM_BOT_TOKEN`
  (or a configured env var name) only. Logging uses `core/redact`'s
  14 patterns (token regex among them) on every audit / pino log
  serializer.
- ❌ **No public listener**. Daemon does not bind to any TCP/UDP port
  (`grammY`'s long-poll is OUTBOUND only; webhooks are off — no
  reverse-tunnel webhook config in Phase 3). Boundary test asserts
  no `net.createServer`/`http.createServer`/`net.Server` in production
  src.
- ❌ **No `/cu` command parsing** in Phase 3 CommandRouter (so a typo
  doesn't accidentally activate a Phase 6 surface that doesn't exist).
  CommandRouter explicitly errors with "Computer Use is not supported
  in this Phase" for any `/cu` / `/computer-use` prefix.
- ❌ **No DB migration script that auto-runs in production without
  the `--auto-migrate` config flag**. Default = false; daemon refuses
  to start if DB schema mismatches and `--auto-migrate` is unset.
- ❌ **No tokens or unredacted approval params in SQLite columns**.
  Audit table stores redacted JSON (already true in `core/audit`'s
  ring buffer; mirror to SQLite via the same redact path).
- ❌ **No `process.exit` from any production module other than the
  CLI/daemon entrypoints**. Supervisor halts via fatal audit; library
  code never exits.

## 7. Architecture decisions D22+

(Phase 1 D5–D10 + Phase 2 D11–D21 carry forward unchanged.)

- **D22** — `SecurityPolicy` is a synchronous `check(input) → Decision`
  function injected into the daemon wire-up. Synchronous because every
  branch can be answered from in-memory tables loaded once at startup
  (allowlist arrays, deny-pattern regexes). Async I/O at decision time
  is forbidden (no DB fetch per inbound message; reload via
  `policy.reload()` triggered by SIGHUP or admin CLI).
- **D23** — `SessionRouter` owns chat→project→thread mapping. Returns
  a `SessionContext` synchronously from in-memory cache; cache writes
  through to SQLite asynchronously. On daemon restart, cache rebuilds
  from SQLite at startup. Cache miss = synchronous DB read on the
  first inbound message after startup; subsequent messages are
  in-memory.
- **D24** — `@codex-im/im-telegram` consumes only
  `@codex-im/channel-core` types + `grammY` + Node stdlib. Boundary
  test asserts no import of `@codex-im/core`, `@codex-im/codex-runtime`,
  `@codex-im/app-server-client`. (Mirrors F13 for channel-core.)
- **D25** — Daemon wire-up is a class (`Daemon`), not a free
  function. Constructor takes a `DaemonOptions` injection bag of
  factories (security policy, session router, channel adapter, broker,
  supervisor, runtime, audit). `start()` and `stop()` are the only
  public methods. Test rigs construct a `Daemon` with the
  `TelegramShapeFakeChannelAdapter` (no separate test rig file).
  `phase2-e2e-rig.ts` stays as historical reference; new tests use
  `Daemon` directly.
- **D26** — `CommandRouter` lives in `@codex-im/core`. Pure function:
  `routeInbound(text) → { kind: "command", name, args } | { kind:
  "prompt", text, attachments }`. CommandRouter is platform-neutral;
  Telegram adapter just passes the text through. Slash commands
  (`/help`, `/projects`, `/new`, `/use`, `/status`, `/stop`) defined
  here.
- **D27** — `@codex-im/storage-sqlite` is a new package. Public surface:
  - `openDatabase(path) → Database` (idempotent, runs migrations iff
    `auto_migrate` opt is set).
  - Repository classes: `BindingRepository`, `ApprovalRepository`,
    `AuditRepository`, `EventLogRepository`. Each is a thin wrapper
    over typed SQL. No ORM.
  - Boundary test asserts no import of `@codex-im/core`,
    `@codex-im/codex-runtime`, `@codex-im/app-server-client`,
    `@codex-im/channel-core`. Storage is below core; core consumes
    storage via dependency injection.
- **D28** — `@codex-im/config` is a new package (or a sub-module of
  daemon). Loads TOML from `~/.codex-im-bridge/config.toml` (path
  configurable via `CODEX_IM_CONFIG`); validates with `zod`; resolves
  env-var references (`${ENV.IM_TELEGRAM_BOT_TOKEN}`); never logs the
  resolved values. (07-SECURITY §8.)
- **D29** — `Daemon.start()` enforces a strict initialization order:
  1. load + validate config; 2. open SQLite + run migrations (if
  enabled); 3. construct broker (placeholder client) + attach; 4.
  construct supervisor; 5. start supervisor (spawns codex);
  6. construct security policy + session router from config; 7. start
  IM adapters (last — so unauthorized inbound messages can't reach
  a half-built daemon); 8. wire subscriptions in the order:
  `broker.onPendingCreated → render → policy.check → adapter.sendCard
   → bindActorPolicy`, `adapter.onAction → policy.check → broker.resolve
   → adapter.answerAction → adapter.updateCard`.
- **D30** — Telegram callback_data encoding: `${approvalId}|${kind}|${nonce}`.
  Same shape as `TelegramShapeFakeChannelAdapter` for parity with
  Phase 2 e2e tests. `approvalId` is `${turn-id-prefix}-${seq}` where
  the seq is bounded so the total payload always fits 62B; the
  encoder throws synchronously on overflow.
- **D31** — Audit ring buffer in `@codex-im/core` continues to be the
  primary audit surface; SQLite is a write-through copy via a
  daemon-level `audit.onEmit(record => sqlite.insert(record))`
  subscriber. No async backpressure: SQLite write is sync (better-sqlite3),
  and a failure logs but does not block the broker.

## 10. Security model

(Tightening 07-SECURITY-AND-COMPUTER-USE.md to Phase 3 scope.)

### 10.1 Inbound message gating

```text
Telegram update
  → adapter.onMessage handler (in @codex-im/im-telegram)
  → Daemon.handleInboundMessage(msg)
    → SecurityPolicy.checkUserAndChat(target, sender)
      ├── deny? → drop silently (no reply, no audit beyond a
      │           "denied unauthorized inbound" record)
      └── allow → CommandRouter.route(msg.text)
            ├── command? → Daemon.handleCommand(...)
            └── prompt?  → SessionRouter.resolve(target)
                  → CodexRuntime.startTurn(...)
```

### 10.2 Approval gating

```text
broker.onPendingCreated(snap)
  → SecurityPolicy.checkApprovalDestination(snap, target)
    ├── deny? → broker.resolve(... decline ...) (synthetic actor =
    │           {kind: "system", reason: "policy_denied"})
    │           do NOT render the card to the user
    └── allow → projectAsRichBlock → adapter.sendCard
          → bindActorPolicy(snap.id, {allowedActors: [user],
                                      target, callbackNonce})
```

`SecurityPolicy.checkApprovalDestination` returns `{kind: "allow"} |
{kind: "auto_decline", reason: string}`. The auto-decline path
records an audit event with the reason but never reaches the IM
layer.

### 10.3 Action gating

```text
adapter.onAction(action)
  → Daemon.handleInboundAction(action)
    → SecurityPolicy.checkUserAndChat(action.target, action.sender)
      ├── deny? → adapter.answerAction(action.callbackHandle, {ok: false,
      │           userMessage: "unauthorized"}) (no broker.resolve)
      └── allow → broker.resolve(action.approvalId, {
            decision: action.uiAction,
            actor: {kind: "im", platform, userId},
            target: action.target,
            callbackNonce: action.callbackNonce,
          })
            → on result.error.kind === "wrong_actor"|"stale_callback"|
              "wrong_target"|"binding_required"|"expired"|... :
                adapter.answerAction({ok:false, userMessage: ...})
            → on ok: adapter.answerAction({ok:true, ...})
                     adapter.updateCard(... terminal state ...)
```

### 10.4 Failure modes (defense-in-depth)

| Attack / mistake | Where caught | Test |
|---|---|---|
| Unauthorized chat sends prompt | SecurityPolicy.checkUserAndChat (1st gate) | T-Sec-1 |
| Authorized chat, unauthorized user | SecurityPolicy.checkUserAndChat (1st gate) | T-Sec-2 |
| Authorized user clicks button on a card belonging to a different chat | SecurityPolicy + broker `wrong_target` | T-Sec-3 |
| Replay of a prior callback (stale nonce) | broker `stale_callback` | T-Sec-4 |
| Click after expiry | broker `expired` (lazy in `resolve`) | T-Sec-5 |
| Click after transport closed | broker `transport_lost` | T-Sec-6 |
| Forged Telegram callback (impossible — adapter only injects what comes from grammY's authenticated handler) | adapter trusts grammY signatures | T-Sec-7 |
| Bot-token leak via log | `core/redact` patterns + pino redact serializer | T-Sec-8 |
| Computer-Use prefix | CommandRouter rejects + audit `command_blocked` | T-Sec-9 |
| Deny-pattern command in user prompt | SecurityPolicy.checkCommand (called inside the broker's onPendingCreated for `command_execution` kind) | T-Sec-10 |

### 10.5 Token handling

- Bot token sourced from `process.env[<envVarName>]` at daemon start.
  If absent, daemon refuses to start with a clear error (no Telegram
  adapter constructed, daemon halts).
- Token never stored in SQLite, never written to logs, never sent
  through any audit event. `redact` patterns include the Telegram
  bot-token shape (`\d+:[A-Za-z0-9_-]{35}`).
- Test asserts that even an intentional `audit.emit({metadata: {token:
  '1234567890:AAaaaa…'}})` is redacted in the ring buffer + the SQLite
  write-through copy.

## 20. Rollback strategy

Phase 3 lands on `phase-3-implementation` branch (separate from this
`phase-3-planning` branch). Each task lands as a separate commit; tag
candidates only apply at the end.

If a regression is found mid-phase:

- **Per-task**: revert the specific commit. Phase 1/2 commits stay
  intact. Re-do.
- **Phase-wide**: if a structural decision (e.g. D29 init order) is
  wrong, branch off `phase-2-approval-im-surface-complete` again,
  cherry-pick the still-good commits, drop the bad ones, re-run all
  reviews.
- **Production launchd uninstall**: `pnpm run launchd:uninstall`
  removes the plist; daemon stops; no other system state changes
  (no daemon-managed system files outside `~/.codex-im-bridge/`).
- **DB migration rollback**: each migration has a `-- DOWN:` comment
  with the SQL to undo it. `pnpm run db:rollback <version>` runs them
  in reverse. Phase 3 ships the runner; running it manually is the
  rollback procedure.
- **Telegram bot lockout**: revoke bot token via @BotFather; rotate
  via `IM_TELEGRAM_BOT_TOKEN` env var; daemon restart picks up the new
  token.

A force-rollback to "no Phase 3" is `git checkout
phase-2-approval-im-surface-complete` + `pnpm run launchd:uninstall`
+ remove SQLite. The Phase 2 tag's surface is untouched; only the
operational surface needs cleanup.
