# v0.1.0-alpha.6 Draft Release Notes

Status: draft; do not publish until the tag is cut.

## Why alpha.6

`v0.1.0-alpha.5` points at `4212ec3`, while `main` now includes `0fb790c`
(`feat(im): default ordinary turns to normal output`). GitHub has no Release
object for `v0.1.0-alpha.5`. Do not move the existing alpha.5 tag; publish
alpha.6 from the new release commit so the customer-facing tag and release
notes match the code customers receive.

## Highlights

- Default IM output is now normal mode: ordinary prompts return the assistant
  answer without Codex status blocks, token usage, item summaries, or automatic
  command log attachments.
- Internal context-loading command noise is suppressed in normal mode.
- Normal IM output redacts local absolute paths such as `/Users/...` into
  customer-safe placeholders.
- `/new <task>` starts a new conversation and submits `<task>` as the first
  prompt; `/new --title <title>` remains the title-only form.
- App Server semantic guard is wired into the contract checks.

## Alpha Limitations

- This is controlled customer alpha, not production.
- `writable_roots` is metadata-only in this alpha when the current pinned Codex
  protocol does not expose an enforceable request field.
- `upgrade --apply` activates the current source checkout and restarts the IM
  daemon; rollback automation is still unavailable, so roll back by checking
  out a previous tag and running `pnpm codex-im:upgrade --apply`.
- Remote-control is status-only / non-goal for this release. Do not expose a
  public remote-control WebSocket.
- Customers must not paste bot tokens, app secrets, callback payloads, private
  keys, cookies, or full daemon logs into support channels.

## Recommended Publish Action

After the R1 refactor gates pass:

```bash
git tag -a v0.1.0-alpha.6 -m "Codex IM Rich Client v0.1.0-alpha.6 customer alpha"
git push origin v0.1.0-alpha.6
gh release create v0.1.0-alpha.6 --title "v0.1.0-alpha.6 customer alpha" --notes-file docs/internal/releases/v0.1.0-alpha.6-draft.md --prerelease
```
